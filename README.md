### 2. Анализ архитектуры монолитного приложения

Приложение SmartHome написано на языке GoLang. В качестве хранилища данных используется Postgres.

Приложение состоит из 2-х компонент:
1. Temperature API - Компонент сбора температуры с датчиков клиента.
   Представляет собой приложение по сбору данных с датчиков непосредственно в доме клиента.
   Предоставляет доступ к показателям датчиков через REST API.
2. Smart Home App - Компонент управления датчиками клиента. Выполняет основные пользовательские функции.
   Представляет собой приложение по сбору данных с датчиков непосредственно в доме клиента.
   Предоставляет доступ к функциям управления датчиками клиента через REST API.
   Осуществляет сбор температуры через REST API установленого у клиента Temperature API.
3. DB - Компонент базы данных хранящий сотояние датчиков и их расположение.

Каждая установка датчиков сопровождается выездом специалиста по подключению датчиков системы отопления в доме.
После установки специалист регистрирует датчики в Smart Home App, состояние которых сохраняется в БД.
Также, специалист настраивает сервис снятия температуры с датчиков (Temperature API) указывая в нем идентификаторы и расположения датчиков клиента.
Для снятия показателей температура датчика клиент запрос в сервис Smart Home App, 
далее система шлет запрос в сервис снятия температуры, который присылает состояние датчика, и значение температуры.
Сервис Smart Home App записывает значение температуры в БД и возвращает значение температуры клиенту.
В случаее недоступности сервиса снятия температуры приложение Smart Home App возвращает последнее записанное значение из БД.

После этого у клиента появляется возможность управлять температурой в доме.

### 3. Определение доменов и границы контекстов

В текущей реализации бизнесс-модели можно выделить 3 домена:
 - Sensor Management: управление состоянием сенсоров мониторинга температуры.
 - Heating Management: управляет изменением температуры в доме (Отсутствует в кодовой базе).
 - Customer Management: управление клиентами, учет установок системы, доступ пользователей (Видимо, реализовано за рамками системы).

Домен Sensor Management можно разбить на контексты:
 - Sensor Registration: регистрации сенсоров в системе.
 - Temperature Monitoring: снятие показателей температуры с датчиков.

Домен Heating Management можно разбить на контексты:
 - Heating Facility Registration: регистрация оборудования нагрева.
 - Heating Facility Control: управлений оборудованием нагрева помещений дома.


### **4. Проблемы монолитного решения**

Текущая реализация поражает множество сложностей, в поддержке и обслуживании клиентов
 - Не предусмотрен запуск одного приложения для обслуживания нескольких клиентов.
   Для каждого клиента устанавливается свой экземпляр приложения.
   Это может вызывать чрезмерное использование ресурсов и затрудняет распределения нагрузки на систему.
 - Отсутствует механизм управления клиентскими приложениями.
   Критически осложняет управление бизнесом.
   Очень сложно осуществлять техническую поддержку существующих клиентов.
   Делает практически невозможным осуществлять мониторинг и снятие метрик использования системы в целом.
 - Каждому клиенту необходимо внедрить приложение для снятия показателей температуры с датчиков.
   Это означает, что сервис должен быть доступен по внешнему статическому адресу со стороны клиента.
   Это удорожает каждую инсталляцию и увеличивает сложность в обслуживании.
 - Существует необходимость устанавливать соответствие сенсоров на стороне клиента и на стороне в настройках приложения (Temperature API).
   В случае потере настроек на стороне клиента, необходим вызов специалиста по восстановлению этих настроек.

# Задание 4. Создание и документирование API

### 1. Тип API

В данной системе можно выбрать гибридное взаимодействие между микросервисами.
Т.к. в требованиях заявлена событийная модель взаимодействия между устройствами и датчиками через сценарии, то для этих целей больше подойдет асинхронный подход.
Синхронный API подходит для непосредственного управления сценариями (создание, изменение, удаление).
Также синхронный API возможен для взаимодействия с клиентом, т.к. в требованиях отсутствует необходимость отслеживания показателей в реальном времени с его стороны.
Если в бушующем подобное требование появится, то клиентский API стоит обогатить асинхронным взаимодействием.

# Задание 5. Работа с docker и docker-compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2) Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081

3) Кроме того для smart_home приложения требуется база данных - добавьте в docker-compose файл настройки для запуска postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры

Ревьюер будет проверять точно так же.
